### 1. Write Model

**Агрегат:** _JobOffer_ (отклик на вакансию)

Write Model представляет собой нормализованную модель для операций записи. Агрегат JobOffer содержит всю бизнес-логику и инварианты системы.

```python
# domain/models/job_offer.py (фрагмент)
from dataclasses import dataclass, field
from typing import List, Optional

@dataclass
class JobOffer:
    id: str
    company: str
    position: str
    user_id: str
    status: str = "active"
    current_stage: str = "resume_sent"
    notes: List[str] = field(default_factory=list)
    events: List[object] = field(default_factory=list)

    @classmethod
    def create(cls, offer_id: str, company: str, position: str, user_id: str) -> "JobOffer":
        offer = cls(id=offer_id, company=company, position=position, user_id=user_id)
        offer.events.append(JobOfferCreatedEvent(offer_id, company, position, user_id))
        return offer

    def change_stage(self, new_stage: str, note: Optional[str] = None) -> None:
        stages = ["resume_sent", "hr_screening", "tech_interview", "test_task", "final_interview", "offer"]
        current_idx = stages.index(self.current_stage)
        new_idx = stages.index(new_stage)
        
        if new_idx <= current_idx:
            raise ValueError("Cannot go back")
        if new_idx > current_idx + 1:
            raise ValueError("Cannot skip stages")
        
        self.current_stage = new_stage
        self.events.append(StageChangedEvent(self.id, stages[current_idx], new_stage, note))
        
        if new_stage == "offer":
            self.status = "offer_received"
            self.events.append(OfferReceivedEvent(self.id, self.company, self.position))

    def mark_rejected(self, reason: Optional[str] = None) -> None:
        if self.status == "offer_received":
            raise ValueError("Cannot reject offer with received offer")
        self.status = "rejected"
        self.events.append(OfferRejectedEvent(self.id, reason))

    def add_note(self, content: str) -> None:
        if not content or not content.strip():
            raise ValueError("Note cannot be empty")
        self.notes.append(content)
        self.events.append(NoteAddedEvent(self.id, content))


**Структура Write Model:**
- `id` — уникальный идентификатор отклика (формат OFFER-YYYY-NNNN)
- `company` — название компании-работодателя
- `position` — должность
- `user_id` — идентификатор пользователя
- `status` — статус отклика (ACTIVE, REJECTED, OFFER_RECEIVED)
- `current_stage` — текущий этап отбора
- `notes` — список заметок
- `events` — список доменных событий

**Инварианты (бизнес-правила):**
- Нельзя перейти на предыдущий этап отбора
- Нельзя пропускать этапы (только последовательно)
- Нельзя изменить отклонённый отклик
- Нельзя изменить отклик с полученным офером
- Заметка не может быть пустой
- При переходе на этап OFFER статус автоматически меняется на OFFER_RECEIVED

**Доменные события:**
- `JobOfferCreatedEvent` — создание отклика
- `StageChangedEvent` — смена этапа
- `OfferRejectedEvent` — отклонение отклика
- `OfferReceivedEvent` — получение офера
- `NoteAddedEvent` — добавление заметки

### 2. Read Model

**Проекция:** _JobOfferView_

Read Model представляет собой денормализованную модель для операций чтения. Данные предварительно агрегируются и оптимизируются для быстрых запросов.


# cqrs/read_model/job_offer_view.py
from dataclasses import dataclass, field
from datetime import datetime
from typing import List

@dataclass
class JobOfferView:
    id: str
    company: str
    position: str
    user_id: str
    status: str = "active"
    current_stage: str = "resume_sent"
    progress_percentage: float = 0.0
    notes: List[str] = field(default_factory=list)
    is_active: bool = True
    is_rejected: bool = False
    is_offer_received: bool = False
    notes_count: int = 0
    created_at: datetime = field(default_factory=datetime.now)
    updated_at: datetime = field(default_factory=datetime.now)
    
    def to_dict(self) -> dict:
        return {
            "id": self.id,
            "company": self.company,
            "position": self.position,
            "status": self.status,
            "current_stage": self.current_stage,
            "progress_percentage": self.progress_percentage,
            "notes_count": self.notes_count
        }


**Структура Read Model:**
- `id` — идентификатор отклика
- `company` — название компании
- `position` — должность
- `user_id` — идентификатор пользователя
- `status` — статус отклика
- `current_stage` — текущий этап
- `progress_percentage` — процент прохождения этапов
- `notes` — список заметок
- `is_active` — флаг активности
- `is_rejected` — флаг отклонения
- `is_offer_received` — флаг получения офера
- `notes_count` — количество заметок
- `created_at` — дата создания
- `updated_at` — дата обновления

**Преимущества Read Model:**
- Денормализованная структура для быстрых запросов
- Заранее рассчитанный прогресс
- Отдельное хранилище для чтения
- Возможность использования материализованных представлений

### 3. Event-Driven Synchronization

**Механизм синхронизации:**

При изменении Write Model (агрегата JobOffer) генерируются доменные события. Эти события публикуются через Event Publisher и обрабатываются Projection, который обновляет Read Model.


# cqrs/projection/job_offer_projection.py
from cqrs.read_model.job_offer_view import JobOfferView

class JobOfferProjection:
    def __init__(self, read_repository):
        self._read_repo = read_repository

    def handle_event(self, event) -> None:
        if isinstance(event, JobOfferCreatedEvent):
            view = JobOfferView(
                id=event.offer_id,
                company=event.company_name,
                position=event.position,
                user_id=event.user_id
            )
            self._read_repo.save(view)
            
        elif isinstance(event, StageChangedEvent):
            view = self._read_repo.find_by_id(event.offer_id)
            if view:
                view.current_stage = event.new_stage
                stages = ["resume_sent", "hr_screening", "tech_interview", "test_task", "final_interview", "offer"]
                current_idx = stages.index(event.new_stage)
                view.progress_percentage = (current_idx / (len(stages) - 1)) * 100
                self._read_repo.save(view)
                
        elif isinstance(event, OfferRejectedEvent):
            view = self._read_repo.find_by_id(event.offer_id)
            if view:
                view.status = "rejected"
                view.is_active = False
                view.is_rejected = True
                self._read_repo.save(view)
                
        elif isinstance(event, OfferReceivedEvent):
            view = self._read_repo.find_by_id(event.offer_id)
            if view:
                view.status = "offer_received"
                view.is_active = False
                view.is_offer_received = True
                view.progress_percentage = 100.0
                self._read_repo.save(view)
                
        elif isinstance(event, NoteAddedEvent):
            view = self._read_repo.find_by_id(event.offer_id)
            if view:
                view.notes.append(event.note_content)
                view.notes_count = len(view.notes)
                self._read_repo.save(view)


**События и их обработка:**

| Событие | Действие в Read Model |
|---------|----------------------|
| `JobOfferCreatedEvent` | Создание записи в JobOfferView |
| `StageChangedEvent` | Обновление current_stage и progress_percentage |
| `OfferRejectedEvent` | Установка статуса rejected и is_active=False |
| `OfferReceivedEvent` | Установка статуса offer_received, is_offer_received=True, progress=100% |
| `NoteAddedEvent` | Добавление заметки в notes, обновление notes_count |

**Компоненты синхронизации:**
- `EventPublisher` — публикация событий
- `JobOfferProjection` — обработка событий и обновление Read Model
- `JobOfferReadRepository` — репозиторий для Read Model

### 4. Оптимизация запросов

Для оптимизации запросов к Read Model используются:


-- Индексы
CREATE INDEX idx_job_offer_views_user_id ON job_offer_views(user_id);
CREATE INDEX idx_job_offer_views_status ON job_offer_views(status);
CREATE INDEX idx_job_offer_views_current_stage ON job_offer_views(current_stage);
CREATE INDEX idx_job_offer_views_user_status ON job_offer_views(user_id, status);

-- Материализованное представление для активных откликов
CREATE MATERIALIZED VIEW mv_active_offers AS
SELECT id, company, position, user_id, current_stage, progress_percentage
FROM job_offer_views
WHERE is_active = TRUE AND is_rejected = FALSE AND is_offer_received = FALSE;

-- Индекс для материализованного представления
CREATE INDEX idx_mv_active_offers_user_id ON mv_active_offers(user_id);

-- Функция обновления
CREATE OR REPLACE FUNCTION refresh_active_offers_mv()
RETURNS TRIGGER AS $$
BEGIN
    REFRESH MATERIALIZED VIEW CONCURRENTLY mv_active_offers;
    RETURN NULL;
END;
$$ LANGUAGE plpgsql;


**Индексы:**
- Индекс по `user_id` — для быстрого поиска откликов пользователя
- Индекс по `status` — для фильтрации по статусу
- Индекс по `current_stage` — для фильтрации по этапу
- Составной индекс `(user_id, status, current_stage)` — для частых фильтров
- Индекс по `created_at DESC` — для сортировки по дате

**Материализованные представления:**
- `mv_active_offers` — представление активных откликов
- `mv_user_offer_stats` — статистика по пользователям

**Триггеры для автоматического обновления:**
- Автоматическое обновление материализованных представлений при изменении данных

## Вывод

В рамках лабораторной работы реализован паттерн CQRS для сервиса «Оффер где?»:

1. **Write Model (JobOffer)** — нормализованный агрегат с инвариантами и доменными событиями. Содержит бизнес-логику управления откликами.

2. **Read Model (JobOfferView)** — денормализованная проекция, оптимизированная для быстрых запросов. Содержит предрассчитанные поля (прогресс, флаги статусов).

3. **Event-Driven синхронизация** — реализована через Event Publisher и Projection. При изменении агрегата публикуются события, которые обновляют Read Model.

4. **Оптимизация запросов** — созданы индексы и материализованные представления для ускорения операций чтения.

Архитектура позволяет разделить ответственность между записью и чтением, упростить бизнес-логику и оптимизировать операции чтения для пользовательского интерфейса и аналитики.
