## Ход выполнения работы

### 1. Протофайлы (.proto)



Созданы файлы `proto/request_service.proto` и `proto/group_service.proto`.

**Сервис RequestService (JobOffer Service):**
- `CreateOffer` — создание отклика
- `GetOffer` — получение отклика по ID
- `ChangeStage` — смена этапа отбора
- `AddNote` — добавление заметки
- `RejectOffer` — отклонение отклика
- `ListActiveOffers` — список активных откликов
- `StreamOfferUpdates` — поток обновлений (server-side streaming)

**Сервис GroupService:**
- `CreateGroup` — создание группы
- `GetGroup` — получение группы по ID
- `AddMember` — добавление участника
- `ListUserGroups` — список групп пользователя

**Типы RPC:**
- Unary — стандартный запрос-ответ
- Server streaming — поток данных от сервера к клиенту

---

### 2. gRPC Server

Реализован сервер на Python с использованием `grpcio`. Сервер запускается на порту 50051 и обрабатывает запросы обоих сервисов.

**Хранилище:**
- In-memory словарь для откликов
- In-memory словарь для групп

**Особенности:**
- Валидация входных данных
- Проверка инвариантов (нельзя пропускать этапы, нельзя вернуться назад)
- Генерация ID в формате OFFER-2024-XXXX и GRP-XXXX

---

### 3. gRPC Client

Реализован клиент для тестирования gRPC сервера. Клиент выполняет последовательные вызовы:

1. Создание отклика через `CreateOffer`
2. Получение созданного отклика через `GetOffer`
3. Смена этапа через `ChangeStage`

---

### 4. Server-Side Streaming

Реализован метод `StreamOfferUpdates`, который позволяет клиенту подписаться на поток обновлений по конкретному пользователю.

**Механизм работы:**
- Клиент отправляет `user_id` в запросе
- Сервер открывает поток и удерживает соединение
- При изменении статуса любого отклика пользователя, сервер отправляет обновление
- Клиент получает обновления в реальном времени

**Сценарий использования:**
- Уведомления о смене этапа
- Оповещения об отклонении отклика
- Уведомления о получении офера

---

### 5. Тестирование

Проверены следующие сценарии:
- Создание отклика с валидными данными
- Получение существующего отклика по ID
- Ошибка при получении несуществующего отклика
- Смена этапа на следующий
- Ошибка при пропуске этапа
- Ошибка при переходе на предыдущий этап

## Контрольные вопросы

**1. В чём преимущество gRPC над REST?**  
gRPC использует бинарный протокол HTTP/2, что обеспечивает более высокую производительность, поддержку streaming и автоматическую генерацию клиентского кода.

**2. Почему Protocol Buffers быстрее JSON?**  
Protobuf сериализует данные в компактный бинарный формат, в то время как JSON использует текстовое представление, которое требует больше места и времени на парсинг.

**3. Зачем нужен streaming в gRPC?**  
Streaming позволяет передавать данные в реальном времени без необходимости повторных запросов. Это критически важно для уведомлений, мониторинга и обновлений в реальном времени.
