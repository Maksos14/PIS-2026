🧩 JobOffer Service — Архитектура сервиса
JobOffer Service — это модульная система, реализующая управление откликами на вакансии в процессе поиска работы.

Сервис построен по принципам гексагональной архитектуры (Hexagonal Architecture / Ports & Adapters), что обеспечивает:

слабую связанность компонентов,

независимость доменной логики от инфраструктуры,

удобство тестирования,

возможность легко заменять адаптеры (БД, API, уведомления).

Архитектура разделена на три ключевых слоя:

🟩 1. Domain Layer — доменная модель
Это сердце системы, где находится бизнес‑логика.
Domain не знает о БД, HTTP, DI, фреймворках — он полностью изолирован.

В JobOffer Service доменные сущности:

JobOffer (Агрегат)
Представляет отклик на вакансию:

offer_id — идентификатор отклика

company — название компании

position — должность

user_id — идентификатор пользователя

status — статус отклика (ACTIVE, REJECTED, OFFER_RECEIVED)

current_stage — текущий этап отбора

Методы:

change_stage(new_stage) — перевести отклик на новый этап

mark_rejected() — отметить как отклонённый

User (Сущность)
Представляет пользователя системы:

user_id — идентификатор

email — электронная почта

full_name — полное имя

job_offer_ids — список ID откликов

Методы:

deactivate() — деактивировать пользователя

change_email(new_email) — сменить email

Project (Сущность)
Представляет кампанию/проект поиска работы (группировка откликов):

project_id — идентификатор проекта

name — название проекта (например, "Весенний поиск 2024")

user_id — владелец проекта

job_offer_ids — список ID откликов в проекте

Методы:

add_job_offer(offer_id) — добавить отклик в проект

pause() — приостановить проект

complete() — завершить проект (оффер получен)

Value Objects
OfferStatus — статус отклика (ACTIVE, REJECTED, OFFER_RECEIVED)

StageType — тип этапа (RESUME_SENT, INTERVIEW, TEST_TASK, OFFER)

Принцип:
Domain не зависит ни от одного внешнего слоя.
Все зависимости направлены внутрь — к доменной модели.

🟦 2. Application Layer — бизнес‑операции (use-cases)
Этот слой определяет что может делать система.
Он связывает доменную модель с внешним миром через порты.

Входящие порты (IN):
CreateJobOfferUseCase — создание отклика на вакансию

GetJobOfferUseCase — получение отклика по ID

Эти интерфейсы вызывают контроллеры, CLI, скрипты — любые клиенты.

Исходящие порты (OUT):
JobOfferRepository — работа с хранилищем откликов

NotificationService — отправка уведомлений (о смене этапа, о напоминаниях)

Application Layer зависит только от интерфейсов, а не от реализаций.

JobOfferService
Реализация входящих портов. Он:

принимает команды от контроллеров,

вызывает доменную модель,

использует исходящие порты для сохранения данных или отправки уведомлений.

Принцип:
Application Layer зависит от Domain, но не от Infrastructure.

🟨 3. Infrastructure Layer — адаптеры и внешние системы
Этот слой содержит реализации портов, то есть всё, что связано с внешним миром:

Входящие адаптеры:
JobOfferController — REST API (пока заглушка / in-memory)

Исходящие адаптеры:
InMemoryJobOfferRepository — временное хранилище в памяти

ConsoleNotificationService — вывод уведомлений в консоль (заглушка)

Конфигурация:
DependencyContainer — DI‑контейнер, который связывает:

сервисы,

репозитории,

контроллеры.

Принцип:
Infrastructure зависит от Application, но не наоборот.

🔄 Потоки данных
text
Клиент вызывает REST‑эндпоинт → JobOfferController
                                    │
                                    ▼
Контроллер вызывает входящий порт → JobOfferService
                                    │
                                    ▼
JobOfferService:
  • создаёт доменные объекты (JobOffer)
  • вызывает исходящие порты (JobOfferRepository, NotificationService)
                                    │
                                    ▼
Infrastructure Layer сохраняет данные или выводит уведомления
🧱 Преимущества такой архитектуры
Преимущество	Описание
Тестируемость	Можно тестировать доменную логику без БД и API
Гибкость	Можно заменить InMemoryRepository на PostgreSQL, не меняя код сервиса
Чистота кода	Domain остаётся чистым и независимым
Масштабируемость	Легко добавлять новые адаптеры: Telegram‑бот, email‑рассылки, cron‑задачи для напоминаний
📌 Итог
JobOffer Service — это хорошо структурированный модуль, построенный по принципам гексагональной архитектуры.
Он разделён на три слоя, каждый из которых выполняет свою роль:

Слой	Роль
Domain	Бизнес‑логика (отклики, пользователи, проекты)
Application	Use-cases (создание, получение, изменение этапов)
Infrastructure	Адаптеры и интеграции (REST, in-memory БД, уведомления)
Такой подход делает систему устойчивой, расширяемой и удобной для дальнейших лабораторных работ.
