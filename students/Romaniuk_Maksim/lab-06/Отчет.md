## Цель работы

Создать комплексную стратегию тестирования (юнит, integration, E2E).

---

## Вариант №32 — Поиск работы «Оффер где?» 📮

**Питч:** Трекай отклики как профи. Приложение помогает пользователю отслеживать процесс поиска работы: фиксировать отклики на вакансии, управлять этапами отбора, добавлять заметки и получать напоминания.

**Ядро домена:** JobOffer, User, Project, OfferStatus, StageType, CompanyName.

---

## Ход выполнения работы

---

### Часть 1. Юнит‑тесты Domain Layer

**Назначение:**  
Проверить корректность работы доменных сущностей, инвариантов и доменных событий без участия инфраструктуры.

**Тестируемые элементы:**
- Агрегат `JobOffer`
- Сущности `User`, `Project`
- Value Objects: `CompanyName`, `OfferStatus`, `StageType`, `ReminderTime`
- Инварианты (смена этапов, отклонение отклика, добавление заметок)
- Доменные события (`JobOfferCreated`, `StageChanged`, `OfferRejected`)

**Примеры тестов:**
- Создание отклика с валидными параметрами
- Ошибка при неверном формате ID
- Смена этапа на следующий
- Ошибка при пропуске этапа
- Ошибка при переходе на предыдущий этап
- Добавление заметки к отклику
- Отклонение отклика
- Достижение этапа OFFER генерирует событие
- Прогресс рассчитывается корректно

**Инструменты:** `pytest`

---

### Часть 2. Юнит‑тесты Application Layer

**Назначение:**  
Проверить корректность работы use‑case сервисов, команд и обработчиков.

**Тестируемые элементы:**
- Команды: `CreateJobOffer`, `ChangeStage`, `AddNote`, `RejectOffer`
- Обработчики команд (Handlers)
- Запросы (Queries)

**Подход:**
- Репозиторий мокается
- Проверяется бизнес‑логика, а не БД
- Проверяется публикация доменных событий

**Примеры тестов:**
- Создание отклика вызывает `repo.save()`
- Смена этапа вызывает публикацию `StageChangedEvent`
- Попытка добавить заметку к отклонённому отклику вызывает ошибку
- Query Handler возвращает DTO

**Инструменты:** `pytest`, `unittest.mock`

---

### Часть 3. Интеграционные тесты

**Назначение:**  
Проверить взаимодействие слоёв: Application → Repository → БД.

**Тестируемые сценарии:**
- Сохранение отклика в БД и последующее чтение
- Поиск всех откликов пользователя
- Поиск только активных откликов
- Обновление существующего отклика
- Удаление отклика
- Поиск отклика по пользователю и компании

**Инструменты:**
- `pytest`
- SQLite (для тестов)
- `DatabaseConfig`

---

### Часть 4. E2E‑тесты

**Назначение:**  
Проверить полный пользовательский сценарий «от начала до конца».

**Основной сценарий:**
1. Создание отклика через REST API
2. Получение отклика по ID
3. Смена этапа на HR скрининг
4. Добавление заметки
5. Смена этапа на техническое интервью
6. Отклонение отклика
7. Проверка финального статуса

**Дополнительные сценарии:**
- Ошибки валидации при создании отклика
- Попытка получить несуществующий отклик
- Невозможность пропустить этапы
- Невозможность изменить отклонённый отклик

**Инструменты:**
- `pytest`
- SQLite (для тестов)
- `JobOfferController`
