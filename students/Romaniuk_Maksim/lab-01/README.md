Основной сценарий: Добавление отклика на вакансию 
Первичный актор: Соискатель
Цель: Зафиксировать новый отклик на вакансию в системе трекинга поиска работы

Предусловия:
•	Соискатель авторизован в системе
•	У соискателя заполнен базовый профиль (имя, контактные данные)
•	Компания, в которую отправляется отклик, существует в справочнике (или будет добавлена)
Основной поток (Happy Path):
1.	Соискатель выбирает опцию "Добавить отклик"
2.	Система отображает форму для заполнения информации об отклике
3.	Соискатель вводит название вакансии и выбирает компанию из справочника
4.	Соискатель указывает дату отклика (по умолчанию - текущая)
5.	Соискатель выбирает способ отклика (сайт компании, hh.ru, LinkedIn, email)
6.	Соискатель может добавить ссылку на вакансию и сопроводительное письмо
7.	Соискатель нажимает кнопку "Сохранить"
8.	Система проверяет корректность введенных данных
9.	Система создает запись об отклике со статусом "Активный"
10.	Система создает первый этап ("Отклик отправлен") в воронке этапов по умолчанию
11.	Система устанавливает напоминание на проверку статуса через 7 дней (если включена опция)
12.	Система отображает сообщение об успешном создании отклика
Постусловия:
•	Новая запись об отклике сохранена в базе данных
•	Отклик привязан к профилю соискателя
•	Создан первый этап в истории отклика
•	Запланировано напоминание (если включено)
•	Статистика соискателя обновлена (добавлен 1 активный отклик)
Альтернативные потоки:
3a. Если компании нет в справочнике:
3a1. Соискатель выбирает опцию "Добавить компанию"
3a2. Система открывает форму быстрого добавления компании
3a3. Соискатель вводит название компании и сайт
3a4. Система сохраняет новую компанию
3a5. Возврат к шагу 3 (выбор компании)
•	5a. Если соискатель хочет добавить несколько откликов подряд:
5a1. Соискатель отмечает чекбокс "Добавить еще один"
5a2. После сохранения система очищает форму, но сохраняет компанию
5a3. Возврат к шагу 3 для ввода следующей вакансии
Исключительные ситуации:
8a. Если обязательные поля не заполнены (название вакансии, компания):
8a1. Система подсвечивает незаполненные поля красным
8a2. Система выводит сообщение "Заполните все обязательные поля"
8a3. Use-case возвращается к шагу 2 (форма редактирования)
•	9a. Если при сохранении произошла ошибка базы данных:
9a1. Система откатывает транзакцию
9a2. Система логирует ошибку
9a3. Система уведомляет пользователя: "Не удалось сохранить отклик. Попробуйте позже"
9a4. Use-case завершается неудачей
•	10a. Если не удалось создать этап по умолчанию:
10a1. Система откатывает создание отклика
10a2. Система логирует критическую ошибку
10a3. Система уведомляет администратора о проблеме с воронкой этапов
10a4. Use-case завершается неудачей
•	4a. Если указана дата в будущем:
4a1. Система предупреждает: "Дата отклика не может быть в будущем"
4a2. Система предлагает исправить дату или использовать текущую
4a3. Возврат к шагу 4

note right
  Все обязательные поля заполнены
end note

Applicant -> UI: Нажать "Сохранить"
UI -> API: POST /api/responses\n{vacancy_title, company_id, date, source, link, cover_letter}
activate API

note over API #FFAAAA
  Начало транзакции
end note

API -> DB: BEGIN TRANSACTION
activate DB

API -> DB: INSERT INTO job_responses\n(id, user_id, vacancy_title, company_id, response_date, source, link, cover_letter, status)
note right
  status = 'ACTIVE'
end note
DB --> API: response_id = 'R-2024-0142'

== Создание первого этапа ==

API -> StageSvc: createInitialStage(response_id='R-2024-0142')
activate StageSvc

StageSvc -> DB: SELECT default_stage_config FROM user_settings\nWHERE user_id = ?
DB --> StageSvc: default_config

StageSvc -> DB: INSERT INTO response_stages\n(id, response_id, stage_name, stage_order, created_at, status)
note right
  stage_name = 'Отклик отправлен'\nstage_order = 1\nstatus = 'COMPLETED'
end note
DB --> StageSvc: OK

StageSvc --> API: stage_created = true
deactivate StageSvc

== Настройка напоминания (опционально) ==

API -> ReminderSvc: scheduleReminder(response_id='R-2024-0142', days=7)
activate ReminderSvc

ReminderSvc -> DB: INSERT INTO reminders\n(id, response_id, reminder_date, reminder_type, status)
note right
  reminder_date = NOW() + 7 days\nreminder_type = 'STATUS_CHECK'\nstatus = 'PENDING'
end note
DB --> ReminderSvc: reminder_id = 'REM-123'

ReminderSvc --> API: reminder_scheduled = true
deactivate ReminderSvc

== Обновление статистики ==

API -> DB: UPDATE user_statistics\nSET active_responses = active_responses + 1\nWHERE user_id = ?
DB --> API: OK

API -> DB: COMMIT TRANSACTION

note over API #AAFFAA
  Транзакция успешно завершена
end note

deactivate DB

API --> UI: 201 Created\n{response_id: "R-2024-0142", status: "ACTIVE"}
deactivate API

UI -> UI: Обновить список откликов
UI --> Applicant: Показать сообщение:\n"Отклик успешно добавлен!"

note over Applicant
  Соискатель видит новый отклик
  в списке со статусом "Активный"
end note

@enduml

Описание потока:
•	Соискатель заполняет форму добавления отклика и отправляет запрос
•	API сервис начинает транзакцию в БД
•	Создается запись об отклике со статусом "ACTIVE"
•	Stage Service создает первый этап ("Отклик отправлен") в воронке этапов
•	Reminder Service настраивает напоминание на проверку статуса через 7 дней
•	Обновляется статистика пользователя (увеличивается счетчик активных откликов)
•	Транзакция фиксируется, пользователь получает подтверждение

Участники:
•	Соискатель – инициатор действия, заполняет форму отклика
•	Web UI – интерфейс для взаимодействия с пользователем
•	Job Tracker API – основной бэкенд-сервис, управляющий откликами
•	Stage Service – микросервис для управления этапами отклика
•	Reminder Service – сервис напоминаний
•	PostgreSQL – база данных для хранения всех записей

== Начало сценария ==

Applicant -> UI: Выбрать "Добавить отклик"
UI --> Applicant: Отобразить форму создания отклика

Applicant -> UI: Заполнить данные и нажать "Сохранить"
UI -> API: POST /api/responses
activate API

note over API #FFAAAA
  Начало транзакции
end note

API -> DB: BEGIN TRANSACTION

StageSvc --> API: ERROR: stage_creation_failed

note over API #FF8888
  Обнаружена ошибка создания этапа
  Инициируется откат транзакции
end note

API -> DB: ROLLBACK TRANSACTION

note over Applicant
  Отклик НЕ создан
  Данные НЕ сохранены
  Пользователь видит ошибку
end note

@enduml
 
Описание потока:
•	Соискатель заполняет форму и отправляет запрос на создание отклика
•	API начинает транзакцию и создает запись об отклике
•	При попытке создать первый этап Stage Service возвращает критическую ошибку (воронка этапов не настроена)
•	API обнаруживает ошибку и инициирует ROLLBACK транзакции
•	Все изменения в БД отменяются, отклик не сохраняется
•	Пользователь получает сообщение об ошибке с предложением повторить позже

4. Gherkin-сценарии
👉 Ссылка на файл: scenarios.feature
Feature: Управление откликами на вакансии
  Как соискатель работы
  Я хочу отслеживать свои отклики на вакансии
  Чтобы контролировать процесс поиска работы

  Background:
    Given пользователь авторизован в системе
    And у пользователя заполнен профиль: "Иван Петров", "+7(999)123-45-67"
    And в справочнике компаний существует компания "Яндекс"

  @happy-path
  Scenario: 01. Успешное добавление отклика (Happy Path)
    When пользователь открывает форму добавления отклика
    And пользователь заполняет поля:
      | Название вакансии | Middle Java Developer |
      | Компания          | Яндекс                |
      | Дата отклика      | 15.03.2024            |
      | Способ отклика    | hh.ru                 |
      | Ссылка            | https://hh.ru/vacancy/123 |
      | Сопроводительное  | Рассмотрите моё резюме |
    And пользователь нажимает кнопку "Сохранить"
    Then система создаёт отклик с ID "R-2024-001"
    And отклик имеет статус "ACTIVE"
    And создаётся первый этап "Отклик отправлен"
    And создаётся напоминание на проверку через 7 дней
    And статистика пользователя обновляется: active_responses +1
    And пользователь видит сообщение "Отклик успешно добавлен"

  @validation @negative
  Scenario: 02. Ошибка: Обязательные поля не заполнены
    When пользователь открывает форму добавления отклика
    And пользователь заполняет только поле "Название вакансии" = "Java Developer"
    And пользователь нажимает кнопку "Сохранить"
    Then система НЕ создаёт отклик
    And система подсвечивает красным обязательное поле "Компания"
    And система показывает сообщение: "Заполните все обязательные поля"
    And транзакция в базе данных не начиналась

  @company @negative
  Scenario: 03. Ошибка: Компания не найдена в справочнике
    Given компании "ТехноСтарт" нет в справочнике
    When пользователь заполняет форму отклика:
      | Название вакансии | Frontend Developer |
      | Компания          | ТехноСтарт         |
    And пользователь нажимает "Сохранить"
    Then система предлагает добавить новую компанию
    When пользователь выбирает "Добавить компанию"
    And пользователь вводит:
      | Название | ТехноСтарт          |
      | Сайт    | https://tehnostart.ru |
    And пользователь подтверждает добавление
    Then система создаёт новую компанию "ТехноСтарт"
    And система создаёт отклик с привязкой к новой компании
    And пользователь видит сообщение об успехе

  @critical @database
  Scenario: 04. Ошибка: Воронка этапов не настроена
    Given у пользователя отсутствует конфигурация воронки этапов по умолчанию
    When пользователь отправляет корректно заполненную форму отклика
    Then система пытается создать первый этап
    And Stage Service возвращает ошибку "stage_creation_failed"
    And система выполняет ROLLBACK транзакции
    And отклик НЕ сохраняется в базе данных
    And пользователь видит сообщение:
      | Не удалось добавить отклик. Обратитесь в поддержку |
    And в логах фиксируется критическая ошибка конфигурации

  @external-service @non-critical
  Scenario: 05. Ошибка: SMS сервис недоступен (некритичная)
    Given SMS сервис временно недоступен
    When пользователь успешно добавляет отклик
    Then отклик сохраняется в базе данных (COMMIT выполнен)
    And система пытается отправить SMS подтверждение
    And SMS сервис возвращает TimeoutException через 10 секунд
    And система логирует ошибку отправки SMS
    And статус отклика остаётся "ACTIVE"
    And пользователь видит сообщение:
      | Отклик добавлен! SMS будет отправлено позже |
    And в очереди повторных попыток создаётся задача на отправку SMS

  @conflict @duplicate
  Scenario: 06. Ошибка: Конфликт данных (дубликат отклика)
    Given пользователь уже откликался на вакансию "Яндекс / Java Developer" 14.03.2024
    When пользователь пытается добавить отклик:
      | Название вакансии | Java Developer |
      | Компания          | Яндекс         |
      | Дата отклика      | 15.03.2024     |
    Then система проверяет наличие дубликатов за последние 30 дней
    And система обнаруживает существующий отклик
    And система НЕ создаёт новый отклик
    And пользователь видит предупреждение:
      | Вы уже откликались на эту вакансию 1 день назад |
    And система предлагает открыть существующий отклик

  @database @timeout
  Scenario: 07. Ошибка: Таймаут базы данных
    Given база данных испытывает высокую нагрузку
    When пользователь отправляет форму создания отклика
    Then API начинает транзакцию
    And при выполнении INSERT в таблицу job_responses происходит таймаут
    And система выполняет ROLLBACK транзакции
    And пользователь получает ошибку 503 Service Unavailable
    And в логах фиксируется "Database timeout"
    And метрика "database.errors.timeout" инкрементируется

  @boundary @validation
  Scenario: 08. Граничный случай: Дата отклика в будущем
    When пользователь заполняет форму отклика
    And указывает дату отклика = "20.03.2024" (завтрашняя дата)
    And пользователь нажимает "Сохранить"
    Then система определяет, что дата в будущем
    And система показывает предупреждение:
      | Дата отклика не может быть в будущем |
    And система предлагает два варианта:
      | Использовать текущую дату |
      | Вернуться к редактированию |
    When пользователь выбирает "Использовать текущую дату"
    Then отклик создаётся с текущей датой (15.03.2024)
    And пользователь видит сообщение об успехе

5. Анализ границ ответственности

5.1. Транзакционные границы
Операция	Синхронная/Асинхронная	Откат при ошибке	Retry-стратегия	Идемпотентность
Создание записи об отклике	Синхронная	Да	Нет	Нет
Создание первого этапа	Синхронная	Да	3 попытки	Да
Настройка напоминания	Синхронная	Да	3 попытки	Да
Обновление статистики	Синхронная	Да	Нет	Да
Отправка SMS	Асинхронная	Нет	5 попыток	Да

Контрольные вопросы
Подготовка к защите:
1. Что такое транзакционная граница? Где она проходит в вашем сценарии?
Транзакционная граница — это логическая область, внутри которой все операции должны быть выполнены полностью (commit) или не выполнены вообще (rollback). В моем сценарии отметки шага дорожной карты транзакционная граница проходит от момента нажатия кнопки "Отметить как выполненный" до момента обновления прогресса и выдачи достижений. В эту транзакцию входят: проверка последовательности, отметка шага, обновление прогресса, проверка достижений. Загрузка ресурсов следующего шага находится ЗА границей транзакции, так как ее неудача не должна откатывать выполнение шага.
2. Почему операция X выбрана синхронной, а Y - асинхронной?
•  Синхронные (отметка шага, обновление прогресса, проверка достижений): потому что пользователь должен сразу видеть результат своих действий. Если прогресс не обновится мгновенно, пользователь подумает, что система не работает.
•  Асинхронные (отправка уведомлений, сохранение в Redis, фоновая синхронизация): потому что они не критичны для основного сценария. Пользователь может получить уведомление о бейдже через 5 секунд, это не влияет на его опыт использования.
3. Как обеспечить идемпотентность при повторных запросах?
Идемпотентность обеспечивается несколькими способами:
•  Проверка статуса шага перед выполнением (если шаг уже выполнен — повторно не отмечаем)
•  Использование idempotency_key, который передается с каждым запросом и проверяется в БД
•  Блокировки на уровне БД (optimistic/pessimistic lock) чтобы два запроса одновременно не обновили один шаг
•  В Redis сохраняем результат выполнения, чтобы при повторном запросе вернуть тот же ответ без изменений
4. Что произойдёт, если внешний сервис вернёт ошибку после частичного выполнения операции?
Зависит от места ошибки:
•  Если ошибка внутри транзакции (например, при обновлении прогресса) — выполняется ROLLBACK, все изменения откатываются, шаг не считается выполненным
•  Если ошибка после COMMIT (например, не загрузились ресурсы следующего шага) — транзакция уже зафиксирована, ошибка логируется, пользователь получает частичный функционал (шаг выполнен, но без материалов)
•  В исключительных случаях (как с Redis) данные сохраняются во временное хранилище и восстанавливаются позже
5. Как система обнаружит, что внешний сервис недоступен?
•  Таймауты — если сервис не отвечает в течение заданного (например, 5 секунд для Resources Service)
•  HTTP коды ответа — получение 5xx (500 Internal Server Error, 503 Service Unavailable)
•  Исключения — Connection refused, Socket timeout, Network unreachable
•  Circuit breaker pattern — если сервис недоступен длительное время, он помечается как "сломанный" и вызовы к нему временно прекращаются
•  Health checks — система мониторинга регулярно проверяет доступность сервисов

6. Какие данные нужно логировать для диагностики сбоев?
•  ID пользователя (user_id) — чтобы знать, кого затронула проблема
•  ID дорожной карты и шага (roadmap_id, step_id) — чтобы понимать, на каком этапе сбой
•  Временные метки (timestamp) — когда произошла ошибка
•  Тип ошибки и stack trace — что именно пошло не так
•  Данные запроса (request payload) — что пытался сделать пользователь
•  Статус транзакции (commit/rollback) — дошел ли запрос до конца
•  Результаты повторных попыток (retry count, success/failure) — пыталась ли система восстановиться
•  Внешние сервисы, которые были недоступны — для мониторинга зависимостей
